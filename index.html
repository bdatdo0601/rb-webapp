What is a Website Application (Web App)?

...A little bit of history

Static Website

In the early age of web development, we need a way display data to our users

thus static website was created, built with HTML/CSS & JavaScript, to serve users a set of defined content

Problem Arise

As the internet become more popular, we want to do even more complex things with our website

  • What if we want to show/serve user different content depends on who they are?
  • What if we want to have more interaction between our users and our content?
  • What if we want to process data & user inputs through the content we provided?

Enter Website Application

Website Application (or Web App) was created to solve these problems

At its core, Web App is simply just a website that is defined through user interaction

Web App enabled website to have dynamic content, along with the capability to process data and user input

How does a Web App operate?

Behind the scenes

Under the hood, A Web App would need to be hosted on a machine with computing power (to process data)

The machine will expose some communication protocol (HTTP) to allow everyone to make requests to the Web App

As users making requests from their browser, the Web App will respond with appropriate content

Web App data architecture

There are a few different approach for processing and displaying data in Web App

  • Component-based Architecture
  • Model-View-Controller Architecture

The Model-View-Controller architecture (MVC)

MVC Architecture

Common software pattern with the aim to modularize application logic

The logic in MVC Web App is separated into 3 parts: Controllers, Models and Views

MVC Architecture

  • Controllers: receive requests from users to update data in models and render views accordingly
  • Models: interact with raw data from multiple source (i.e: database, external sources, etc)
  • Views: display content to users based on responses from controller and data from models

Rails Web App

Directory Structure


            ├── app/                 # Application Logic
            │   ├── assets/               # Resources for the applications (styling, images, etc)
            │   ├── controllers/          # Controller classes, handle web requests from users
            │   ├── helpers/              # Helper classes, assist function for controller, model and views
            │   ├── models/               # Model classes, interacted with database/external sources
            │   └── views/                # View template, combine with data to generate HTML for user's browsers
            ├── config/              # Configuration for the application (routing, database settings, etc)     
            ├── db/                  # Database for the application
            ├── Gemfile              # Application's dependencies declaration
            ├── lib/                 # Application's dependencies
            ├── log/                 # Logging errors for application
            ├── public/              # Static content of the application
            ├── Rakefile             # Application build instruction
            ├── test/                # Automated testing for the application
            └── vendor/              # Third party's library

Best Practices in Rails

  • DRY (Don't Repeat Yourself)
  • Strictly enforce your architecture
  • Utilize helpers to keep views, and controllers cleaner

Summary

Web Application are website that can provide dynamic content based user interaction

MVC architecture allows web application logic to be separated, making it easier to develop and scale

Rails helps develop Web App with MVC architecture much faster and easier
